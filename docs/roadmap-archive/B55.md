# B55 — glass/metro-climate: no "Auto" button for Homematic thermostats (falsy-zero mode filter) ✅ fixed

*Archived roadmap item — Bugs. Open items: [../ROADMAP.md](../ROADMAP.md) · Index: [README.md](README.md)*

**✅ Fixed (07/2026)** — the sanitize filter in `_parsedModes` now says what it means (`m && m.value !== null && m.value !== undefined && m.value !== ''`) in glass and metro, and **material got the identical filter too** (it previously had none — the E114 alignment this item asked for). A `{value: 0, label: 'Auto'}` entry survives in all three families and its read-back (`0`) marks the chip/button active. Browser-tested with a value-0 entry in each family. Part of the B53–B58 climate fix wave; the glass state-line sibling defect is B57.

Original item:

Homematic thermostats show no **Auto** mode button on the climate cards — Manu/Off/Boost render, Auto is missing.

**Root cause (confirmed by code read): a falsy-zero filter.** The Homematic discovery stamps Auto as `{value: 0, label: 'Auto', …}` (`CONTROL_MODE`/`SET_POINT_MODE` 0 IS auto, for both generations — [native-discovery.js `buildHmModes`](../../server/src/mqtt/native-discovery.js)). `glass-climate` and `metro-climate` both sanitize the parsed modes list with **`.filter(m => m.value)`** — `0` is falsy, so **the Auto entry is silently dropped** before rendering. `material-climate` has no such filter and is unaffected (its `_parsedModes` maps without filtering).

**Fix:** the filter means "drop empty/invalid entries" — make it say that: `.filter(m => m && m.value != null && m.value !== '')` in both families. Everything downstream already handles value 0 correctly (the active-entry matching compares `String(m.value) === String(v)`).

**Also:** the filter divergence itself (two families sanitize, one doesn't) is E114 drift — align all three on the same corrected filter, and add a unit/browser test with a `value: 0` mode entry so a truthiness regression can't ship again. If Auto is ALSO missing on a **material** card somewhere, that instance's deployed `modes` config predates the discovery stamping — re-apply discovery (same caveat as B53's repro note).

**Acceptance:** a discovery-wired Homematic thermostat shows Auto/Manu/Off/Boost on all three climate cards; tapping Auto publishes `CONTROL_MODE: 0` (BidCoS: `AUTO_MODE`), and an Auto read-back (mode 0) marks the Auto chip active.

**Relates:** E108 ✅ (the discovery that stamps `value: 0`), E102 ✅ (mode-chip machinery), **E114** (parity — the filter must be identical in all three), **B53/B54** (the concurrent climate-mode display bugs — one climate-mode fix wave).
