# B54 — *-climate: active boost is not detected from the device — UI falls back to "Manu" moments after activation ✅ fixed

*Archived roadmap item — Bugs. Open items: [../ROADMAP.md](../ROADMAP.md) · Index: [README.md](README.md)*

**✅ Fixed (07/2026)** — as designed below, all three families:
- New contract attributes `subscribe-boost-state` + `message-property-boost-state` (identical names/help across material/glass/metro, E114). While the topic reads truthy (`true`/`1`), a `_boostForced` flag keeps the momentary (boost) entry displayed active **regardless of the mode read-back** and starts the countdown; a falsy read-back (or tap-off) returns the display to the mode-derived state. Boost started at the wall dial is now visible, and a reload mid-boost shows boost active (client timer restarts from full — accepted E102 degradation).
- The hm climate recognizer stamps `boost_state_topic: hm/status/<ch>/BOOST_MODE` + `message_property_boost_state: payload.val` for **both** generations (BidCoS redundancy is harmless); all three climate `discovery.map`s consume the keys.
- Publishing/off-strategies untouched — the state topic is read-only detection.

Browser-tested (forced-active vs Manu read-back, wall-dial activation, tap-off while forced) in all three families; recognizer stamping unit-tested in `server/test/native-discovery.test.js`. Part of the B53–B58 climate fix wave.

Original item:

Activating boost from feezal works (publish fires, button lights up, the E102 client countdown starts) — but a few moments later the UI flips back to **Manu**. Real-world confirmation that the device DOES report the state: `hm/status/<channel>:1/BOOST_MODE {"val": true, …}` while boost runs.

**Root cause — boost active-state is client-side only; the device's BOOST_MODE status topic is never subscribed.** What exists today:

- The active mode chip derives from `mode_state_topic` — **HmIP: `SET_POINT_MODE`, which keeps reporting 1 (Manu) during boost**. Its first read-back after activation overwrites the client-side active state → the observed flip. (BidCoS `CONTROL_MODE` genuinely reports 3 during boost, so classic devices may detect correctly via the mode topic — verify.)
- The only boost-related subscription is the optional **`subscribe-boost-remaining`** (E102 countdown display, a *different* datapoint) — there is **no `subscribe-boost-state`** anywhere in the three climate elements, and the hm recognizer stamps no BOOST_MODE status topic.

**Fix direction:**
1. **Element contract (all three families, E114):** new `subscribe-boost-state` (+ `message-property-boost-state` twin, payloads `true`/`false` per the hm JSON-Extended `val`). While it reads truthy, the momentary (boost) mode entry is **forced active regardless of the mode read-back**; on falsy, display returns to the mode-topic-derived state. Bonus this buys for free: boost started **at the wall dial** becomes visible in feezal, and the E102 countdown can key off the device-reported transition instead of only the local click (a reload during boost then shows the true state instead of nothing).
2. **Discovery:** the hm climate recognizer stamps `boost_state_topic: hm/status/<ch>/BOOST_MODE` (BOOST_MODE is already in the recognizer's datapoint whitelist — only the config key is missing); map key → `subscribe-boost-state` in the three climate `discovery.map`s. Stamp for both generations; if BidCoS proves redundant (CONTROL_MODE=3 already detects), it's harmless redundancy, not a conflict.
3. **Interaction with deactivate:** the momentary entry's existing `off` strategies (HmIP `BOOST_MODE=false`, BidCoS `restore`) stay untouched — the state topic is read-only detection; publishing remains as E102 built it.

**Acceptance:** activate boost from feezal → chip stays active for the full boost duration (no flip to Manu), countdown runs; activate boost at the device → feezal shows it; boost ends (timeout or deactivate) → display returns to the real mode; page reload mid-boost shows boost active.

**Relates:** E102 ✅ (boost machinery — momentary entries, countdown, off-strategies; this adds the missing state leg), E108 ✅ (hm climate recognizer — BOOST_MODE already whitelisted for accumulation), **E114** (three families, one contract), **B53** (the sibling mode-display bug — same read-back-overwrites-reality mechanism, fix together).
