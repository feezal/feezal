# E131 — Homematic motion detector discovery → *-motion/occupancy elements ✅ done

*Archived roadmap item — Element Ecosystem. Open items: [../ROADMAP.md](../ROADMAP.md) · Index: [README.md](README.md)*

**✅ Implemented (07/2026)** — as one **generalized boolean-sensor recognizer** (`hm-sensor:` ids in [native-discovery.js](../../server/src/mqtt/native-discovery.js)), which also serves E132's hazard classes via a channelType→class table (the datapoints turned out uniform enough — one boolean state datapoint per channel):

- **Table:** `MOTION_DETECTOR` (BidCoS, **confirmed**) and `MOTIONDETECTOR_TRANSCEIVER` (HmIP, best-knowledge) → `device_class: motion`, datapoint `MOTION`; `WATER_DETECTION_TRANSMITTER` → `moisture` (ALARMSTATE/WATERLEVEL_DETECTED/MOISTURE_DETECTED, first observed wins); `SMOKE_DETECTOR` → `smoke` (BidCoS `STATE`). Allowlist gating is safe-by-construction — an unverified name can only miss a device, never false-positive; HmIP-SWSD's alarm-status **enum** is deliberately not wired (equality payload matching can't express it — future work with real metadata). `ILLUMINATION` etc. never confuse the match (datapoint whitelist). The state datapoint must be **observed**, not constructed.
- **Emits** HA-shaped `binary_sensor` config (`device_class`, `state_topic`, payloads `1`/`0` per the contact convention, `value_template '{{ value_json.val }}'`), `:0 UNREACH` availability — and **`battery_low_normalized` from day one, presence-checked per E124**: only when the device was seen publishing `LOWBAT` (BidCoS) or `LOW_BAT` (HmIP) on its `:0` channel; battery is never folded into availability.
- Contact/switch `STATE` and metadata-less channels never promote here (independent channelType gating, framework rule).

Unit tests: promotion + full config, both battery generation names, presence check, hazard classes, non-promotion paths. TESTING.md discovery row with the explicit verify-on-real-hardware caveats. The consuming cards' generalized type list is E132.

Original item:

The occupancy/motion cards (`material-motion`, `glass-occupancy`, `metro-occupancy`) already carry HA discovery (`component: 'binary_sensor'` with a `device_class` → `type` map), so zigbee2mqtt motion sensors auto-wire today — **Homematic motion detectors don't**: there is no native recognizer for them.

**New recognizer** following the contact pattern: match `hm/status/<seg>/MOTION`; channel-type gate `MOTION_DETECTOR` (BidCoS, confirmed), HmIP naming unverified (probably `MOTION` too, channelType may be `MOTIONDETECTOR_TRANSCEIVER` — confirm against a real device's `hm` payload metadata before hardcoding); metadata-less channels never promoted. Emit `component: 'binary_sensor'` with `device_class: 'motion'`, `state_topic` = the MOTION status topic, payloads `true`/`false`, `value_template '{{ value_json.val }}'`, availability from `:0 UNREACH` — and `battery_low_normalized` from day one (`LOWBAT`/`LOW_BAT` on `:0`, presence-checked per E124). Whitelist MOTION only (ILLUMINATION must not confuse the match).

**Relates:** E108 ✅ (recognizer framework — contact recognizer as the closest pattern), **E124** (battery indicator — this recognizer must emit its record from day one), E126 ✅ (switch recognizer — the most recent recognizer addition to crib from), **E132** (the consuming cards generalize to `*-sensor` — implemented against the generalized type list), material-motion / glass-occupancy / metro-occupancy (the consuming cards).
