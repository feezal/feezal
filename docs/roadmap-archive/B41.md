# B41 — Deeplinking to a layout-app sub-view does not work ✅ fixed

*Archived roadmap item — Bugs. Open items: [../ROADMAP.md](../ROADMAP.md) · Index: [README.md](README.md)*

**✅ Fixed (07/2026)** — root cause was neither of the two suspects: the cold-load path worked, but a hash-only URL change never reloads the page — it fires `hashchange`, and `feezal-app-viewer._onHashChange` used `viewFromHash()`, which drops the embedded segment entirely. It now routes the full path through `feezal.site.applyControlCommand('view', 'view/embedded')` (the N30 contract, same as the MQTT view command), so deep links typed into an open viewer, anchors and back/forward all land on the sub-view; a bare sub-view name is offered to the visible shell first. `layout-app.routeToEmbedded()` gained a same-view guard so the drawer's own hash write doesn't re-clone the view it just embedded (package 3.0.8). New unit tests incl. percent-encoded umlauts in both segments. Original report:

Opening a URL that targets a specific `layout-app` sub-view does not land on that view. N30 added view-router delegation and hash `#/<view>/<embedded>` parsing, so the routing contract exists — this is either the app shell not consuming the initial hash on first render (only reacting to *subsequent* changes), or a race where the router resolves before the shell's views are registered.

**Relates:** N30 (view-router registry + `viewPathFromHash()`), B40 (same first-load lifecycle window — check together).
