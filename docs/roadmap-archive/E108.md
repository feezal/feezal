# E108 ‚Äî Native device self-discovery (Homematic climate + WLED) via server-side topic recognizers ‚úÖ implemented (live-system smoke test pending)

*Archived roadmap item ‚Äî Element Ecosystem. Open items: [../ROADMAP.md](../ROADMAP.md) ¬∑ Index: [README.md](README.md)*

Shipped 07/2026. Device families that don't emit Home Assistant discovery configs ‚Äî **hm2mqtt / RedMatic climate** and **WLED** ‚Äî now get the **same one-click Auto-configure UX** as HA lights/covers, by recognizing them **server-side** from the broker firehose the bridge already subscribes to (`#`). Zero editor-UX change: the recognizers synthesize the same discovery-entity shape the HA path produces, so the existing ‚ö° device-picker / "Auto-configure" banner / `_applyDiscovery` consume them unchanged.

- **Server** (`server/src/mqtt/native-discovery.js`, wired into `discovery.js`): a generic `{match, accumulate, promote}` recognizer framework (native path isolated in try/catch so it can never affect HA discovery; `getDiscoveredEntities()` merges native entities with namespaced ids). **Homematic-climate** watches `hm/status/<channelName>/<DP>`, accumulates per-channel datapoint sets, and promotes on signature ‚Äî `SET_TEMPERATURE`+`CONTROL_MODE` ‚Üí BidCoS, `SET_POINT_TEMPERATURE`+`SET_POINT_MODE` ‚Üí HmIP; `VALVE_STATE`/`LEVEL` ‚Üí TRV ‚Äî synthesizing an **HA-keyed climate config** (so it reuses the elements' existing `discovery.map`) plus `message_property:'val'` (MQTT-Smarthome JSON-Extended) and `valve_min`/`valve_max` (1 for HmIP LEVEL, else 100). The per-generation mode arrays (Auto/Manu/Off + `match-setpoint-max`/momentary Boost; HmIP putParamset `VALUES`) are built from the observed prefix+channelName by a CJS port of the former `climate-profiles.js`. **WLED** watches `wled/<id>/v` + the retained `wled/<id>/status` LWT and synthesizes a `wled` entity with `availability_normalized`.
- **Client**: the E102 WP3 **profile-stamping picker was removed** (`feezal-climate-profiles` + `climate-profiles.js` + the `type:'custom'` descriptors/embed) ‚Äî discovery replaces it. The generic **U39 `type:'custom'` inspector hook is kept** (reusable platform infra). The three `*-climate` `discovery.map`s gained the three native keys (`message_property`/`valve_min`/`valve_max`, HA-absent so HA/z2m entities skip them); the three `*-wled` elements gained `discovery:{component:'wled', map:{device_topic‚Üítopic, name‚Üílabel}}` (availability applied automatically from `availability_normalized`). Fixed a latent metro-climate bug along the way: its `modes` map shorthand would have stamped a rich native array as `[object Object]`.

**Confirmed topic structure (RedMatic / hm2mqtt defaults, 07/2026):** read `hm/status/<channelName>/<DP>`, write `hm/set/<channelName>/<DP>`, putParamset `hm/paramset/<channelName>/VALUES`; prefix `hm` (a recognizer constant); payload **MQTT-Smarthome "JSON Extended"** ‚Üí value at `.val`.

Server suite 376 + full unit 772 + full browser 552 green; a client discovery test drives `_applyDiscovery` with native-shaped climate + WLED configs to prove the client map matches the server contract.

**Post-live-test fixes (07/2026):** live testing surfaced and fixed: the Shoelace `<sl-select>` space-in-option-value bug (native `discovery_id`s never resolved on pick ‚Äî encode/decode + a space-free device-address id); **rebuilt the recognizer to drive off the CCU metadata** in each JSON-Extended `hm` object ‚Äî control channel by exact `channelType` allowlist (`HEATING_CLIMATECONTROL_TRANSCEIVER` HmIP / `CLIMATECONTROL_RT_TRANSCEIVER` BidCoS, + a datapoint-signature fallback), grouping by `hm.device`, `hm: <deviceName>` label, valve only when `LEVEL`/`VALVE_STATE` seen, min/max from `datapointMin`/`Max`; `message_property` corrected to **`payload.val`** (feezal paths start at the message root); and **Homematic availability now auto-wired** from the `:0` maintenance `UNREACH` datapoint (`payload-available=false`/`payload-unavailable=true`) ‚Äî which also fixed a latent bug where `availability_normalized` sat at the entity level instead of inside `config` (WLED availability never actually applied either; now fixed for both).

**Deferred / needs live-system confirmation:** (a) a **real-broker smoke test** that discovery finds all thermostats (retained-ness of `hm/status` topics ‚Äî believed yes per mqtt-smarthome); (b) optional `subscribe-boost-remaining` device-datapoint wiring (`BOOST_STATE`/`BOOST_TIME` names/units) ‚Äî the client boost countdown works regardless; (c) one entry per physical device is surfaced (a wall thermostat + TRV + `HmIP-HEATING` group each appear separately ‚Äî legitimately distinct controllable devices; a "prefer group / hide members" filter is a possible future refinement).

**Relates:** E102 (replaced its WP3 manual picker; reused its `climate-profiles.js` knowledge as the server builder; the topic-shape gate noted there was the key input, now confirmed), E103 (WLED elements ‚Äî the discovery targets), N31 (availability normalization ‚Äî reused for WLED; HM availability deferred), the existing server discovery machinery (`discovery.js` + `/api/discovery/*` + the generic `_applyDiscovery` client path ‚Äî all reused unchanged).
