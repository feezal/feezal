# E115 — Switch an element to another family (context menu) ✅ implemented

*Archived roadmap item — Editor UX. Open items: [../ROADMAP.md](../ROADMAP.md) · Index: [README.md](README.md)*

**✅ Implemented (07/2026)** — right-click an element (or a multi-selection) → **Switch family…** → a submenu of the installed families that have a twin for the element's function (the numeric readout pairs across the `*-value` ↔ `eink-number` tag divergence via a normalisation alias). Picking a family swaps each selected element to its twin **in place** (preserving DOM order/grouping): the shared attribute contract (the intersection of the source's attributes with the target's declared attribute names — handling both `{name}` objects and bare-string descriptors, the bug that would otherwise drop metro's `subscribe`), `discovery-id`, `locked` and geometry (`top`/`left`/`width`/`height` + generic inline CSS) carry across; the source family's `--feezal-*` chrome overrides and any orphan attributes are dropped, and a transient report names exactly what was dropped/skipped (never silently lossy). Elements with no twin in the chosen family are skipped and counted; the whole batch is one undo entry and the swapped nodes are re-selected. Function pairing uses the `feezal-element-<family>-<function>` tag convention (E138-standardised) with the `number→value` alias; views/component instances are never offered. Built in `feezal-sidebar-inspector.js`; browser-tested in `test-browser/feezal-switch-family.test.js`. Original item:

Right-click an element → **Switch family** → `metro-light` becomes `glass-light`, **keeping all shared configuration** (topics, payloads, limits, label) and its position/size. Today changing your mind about a family means deleting and re-configuring every element.

**Depends on E114** — parity is what makes the attribute set transferable; without it this is a lossy guess.

#### Function-identity: how the switcher finds the target twin (decided)

The pairing key is **not** string-munging the tag. An element's *function* is already declared by:
- its `discovery.component` (`light`, `climate`, `cover`, `contact`, `sensor`, …) — set on every device-family element; and, post-**E137**, the **shared behavior controller** it consumes (`LightController`, `ClimateController`, …) which is the strongest same-function signal.
- the `feezal-element-<family>-<function>` **tag convention** as the fallback for elements without a controller/discovery.

The switcher enumerates registered custom elements, groups them by (controller ∨ `discovery.component` ∨ tag-function), and offers the **installed** members of the same function group in other families. The `feezal-element-<family>-` prefix (already used by A23's missing-element map) identifies the family facet.

#### Decided behaviour (07/2026)

- **Scope: single element + multi-selection.** Right-click one element, or a multi-selection, → **Switch family → <family>**. Each selected element switches to its twin in the chosen family independently; elements with no twin in that family are skipped (and reported). **Site-wide / per-view "restyle this dashboard"** (switch *all* of family X) is a **documented later tier** — ships after the per-element mapping is proven.
- **Orphan attributes: drop + warn.** Attributes with no counterpart in the target's shared contract are **stripped**, and a summary warns exactly which were dropped on which elements (so the switch isn't silently lossy). Shared-contract attributes, `discovery-id`, and geometry always carry across.
- **Styles: reset to the target family's defaults.** The source family's per-element style-prop overrides (`--feezal-*` colour/size vars, family chrome) are **dropped** — adopting the target's native look is the entire point of restyling. **Geometry (`top`/`left`/`width`/`height`) and generic CSS are preserved**; the target's `defaultStyle` fills any size the element type needs (e.g. a family whose card wants a different min size).
- **Missing target families: not offered.** The Switch-family menu lists **only installed families** that have a twin for the element's function. Installing another family first (packages sidebar) then makes it appear — no inline install flow in this item (keeps the menu honest; revisit if A23's install path makes inline trivial).
- **One undoable action.** The whole switch (tag swap + attribute filter + style reset, across the whole selection) routes through the normal change/undo pipeline as a single history entry.

#### Mechanics

Per element: create the target tag, copy the **intersection** of source-attrs and target's declared attribute names (the shared contract), copy geometry + `discovery-id`, drop everything else, replace the node in place (preserving DOM order / `data-group`), re-select. **E87** ✅'s attribute-remap pattern applies where a shared attribute was *renamed* between families (rare, but the remap table is the place for it). Editor-only; the switched HTML is what the viewer/export consume.

**Open / at-implementation:** the exact "same function" grouping precedence when an element has *both* a controller and a divergent `discovery.component` (controller wins); whether to show a count in the menu (`Switch family → Glass (6 elements)`); the multi-select report UI for skipped/dropped (toast vs. dialog — dialog if any attribute was dropped, toast if clean).

**Relates:** **E114** (parity — precondition; the switcher is only as lossless as the shared contract is complete), **E137** ✅ (behavior controllers — the strongest function-identity signal and what makes the attribute intersection principled), **E113** (function × style — the model that makes this coherent), E87 ✅ (attribute-remap precedent for renamed shared attrs), A23/A24 (tag→package map + family-prefix identification), E138 ✅ (the contact/sensor/motion/value split — switch groups must respect the post-split function boundaries). *(E115's "strip the styling" switch target is the **Basic** family where a functional twin exists — E116 ❌ dropped.)*
