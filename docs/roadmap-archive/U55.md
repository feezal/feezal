# U55 — View tab bar: hold-to-drag so clicks stop reordering views ✅ done

*Archived roadmap item — Editor UX. Open items: [../ROADMAP.md](../ROADMAP.md) · Index: [README.md](README.md)*

**✅ Implemented (07/2026)** — both halves, exactly per the design:

1. **Hold-to-drag arming:** view and folder tabs render `draggable="false"`; `pointerdown` (primary button) starts a 300 ms timer, and only when it fires with the button still down does the tab become draggable — with a visible lift cue (`.drag-armed`: slight scale + shadow, grab cursor). `pointerup`/`pointerleave`/`pointercancel` disarm (the armed state survives while a native drag is in flight; `dragend` resets). The existing `dragstart`/`dragover`/`drop`/`dragend` machinery is untouched — only the arming changed, identically for view and folder tabs. Click, dblclick (rename), context menus unaffected; touch long-press-then-drag is the natural idiom.
2. **Self-drop bug fixed, two independent hardenings in `_applyDrop`:** (a) dropping an item onto ITSELF (before/after itself — the pointer never left the tab) bails **before** detaching (order-preserving by definition; previously the detach-then-locate sequence missed the just-detached node and the fallback appended it to the END — the `view1,view2,view3 → view2,view3,view1` micro-drag repro); (b) the `!loc → push(dragged)` fallback is now an **abort without applying** (matching the existing silent abort for a folder dropped into its own subtree), so no unresolved target can ever mean "move to end".

Canvas element dragging (interact.js) untouched, as decided. Unit tests (self-drop no-op for views and folders, unresolved-target abort, genuine reorder, bar-append) + e2e (jittery click switches without reordering, hold arms/release disarms with the lift cue, self-drop leaves order unchanged, genuine reorder, existing dblclick-rename test passing against the new handlers). TESTING.md row added.

Original item:

Reordering views by accident happens far too often: a click on a view tab that moves a few pixels turns into a drag, and the view (or folder) lands somewhere else in the tab order. The user only wanted to switch views.

**Why it happens:** the tab bar uses **native HTML5 drag** — `.ftab` elements carry `draggable="true"` with `dragstart`/`dragover`/`drop` handlers, and the browser's built-in drag threshold is only a few pixels with **no configurable delay**. Any slightly sloppy click qualifies.

**Wanted: hold-to-drag.** Dragging only arms after the mouse button has been held down for a moment (~250–300 ms) on the tab; a quick click — even a slightly moving one — always stays a click.

**Bundled bug — a self-drop reorders to the END instead of no-op.** Repro: views `view1, view2, view3`; drag `view1` a few px right — the blue drop line sits between view1 and view2 (a no-op position) — release: the order becomes `view2, view3, view1`. Cause: when the drop target IS the dragged tab, `_applyDrop` first detaches the dragged node, then fails to `_locate` that same just-detached node, and the `if (!loc) { tree.push(dragged); }` fallback appends to the end.

**Explicitly out of scope (decided):** canvas element dragging stays exactly as it is (interact.js, separate path). Only the tab bar changes.

**Relates:** U8 ✅ (view folders — the second draggable tab kind, same `_applyDrop` path), U33 (drag/reorder conventions), B33 (the other "pointer gesture misfires" family — closed can-not-reproduce).
