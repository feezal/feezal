# B53 — *-climate: Homematic "Off" state shows as "Manu 4.5°" instead of "Off" without a setpoint ✅ fixed

*Archived roadmap item — Bugs. Open items: [../ROADMAP.md](../ROADMAP.md) · Index: [README.md](README.md)*

**✅ Fixed (07/2026)** — two changes, all three families:
1. **The matcher compared the wrong setpoint:** `_activeModeEntry`'s cap check used `_lastRealSetpoint ?? _setpoint` — so once a real setpoint (21°) had been remembered, the Off read-back (4.5) never matched (`21 <= 4.5`). It now compares the **current** setpoint read-back (falling back to lastReal). This was the actual mechanism behind the user report even on material/glass; metro's "missing matcher" from the original diagnosis had meanwhile landed with E102.
2. **Display suppression:** while the active entry carries `match-setpoint-max` (off-by-setpoint sentinel), the target-temperature display is suppressed — material shows the mode label in the arc centre and parks the fill arc; glass shows the label in the state line and the popup pill; metro shows it on the front line and in the back stepper readout. The actual temperature stays visible, and interacting with the setpoint still publishes (re-entering Manu — correct Homematic semantics).

Browser-tested in all three families (`test-browser/feezal-element-{material,glass,metro}-climate.test.js`), including the previously-remembered-setpoint case. Part of the B53–B58 climate fix wave.

Original item:

Homematic thermostats have no real Off mode — "Off" is **Manu with the 4.5 °C frost-protection setpoint**. Setting Off from feezal works (the discovery-stamped mode entry publishes `CONTROL_MODE: 1, SET_POINT_TEMPERATURE: 4.5`), but the device then reads back mode = Manu + setpoint = 4.5, and the card shows **"Manu" with a 4.5° target** instead of **"Off" with no target temperature**.

**Half of this already exists — and one family is missing it.** E102 ✅ built exactly the needed detector: mode entries may carry `match-setpoint-max` (the hm discovery stamps 4.5 on the Off entry), and the active-chip logic prefers the capped entry while the effective setpoint is ≤ the cap — "Off wins over Manu at ≤ 4.5" ([feezal-element-circle-climate.js](../../www/packages/@feezal/feezal-element-circle-climate/feezal-element-circle-climate.js), same in glass-climate). So:

1. **metro-climate never implemented the matcher** — no `match-setpoint-max` handling exists in it at all (grep-verified); its back-side mode chips show Manu in the Off state. Port the E102 logic (E114 parity gap). *(Resolved before this fix landed — E102's port had reached metro in the meantime.)*
2. **The setpoint display ignores the matched Off state in ALL three families** — even where the chip correctly says "Off", the arc/readout still renders the 4.5° target. New behaviour: when the ACTIVE mode entry carries `match-setpoint-max` (i.e. it is an off-by-setpoint sentinel state), **suppress the target-temperature display** — show the mode label in the centre instead of "→ 4.5°", grey/park the setpoint arc; the actual temperature stays visible. Interacting with the setpoint (drag/step) while Off naturally publishes a new setpoint and thereby re-enters Manu — that is correct Homematic semantics and should keep working exactly so.
3. **Repro caveat for the user's report:** if the chip *also* says "Manu" on a material/glass card, the deployed element's `modes` config may predate the E102 stamping (no `match-setpoint-max` on its Off entry) — re-applying discovery refreshes it; the TESTING row should cover both a fresh and a re-applied config.

**Acceptance:** hm thermostat set to Off (via feezal or the wall dial): all three climate cards show **"Off"** as the active mode with **no target temperature displayed**; raising the setpoint (dial or drag) flips the display back to Manu + target; non-Homematic climates (no `match-setpoint-max` entries) are pixel-identical to today.

**Relates:** E102 ✅ (the `match-setpoint-max` matcher — this completes its display story), E108 ✅ (hm climate discovery stamps the Off entry), **E114** (parity — metro missing the matcher is exactly the drift the contract exists for), B37 ✅ (climate card geometry — where the centre readout lives), **B54** (boost detection — the sibling "mode read-back overwrites reality" bug; fix together).
