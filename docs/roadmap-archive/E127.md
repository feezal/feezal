# E127 тАФ Homematic dimmers: settling behaviour (`WORKING` / `_NOTWORKING`) тАФ no slider jumping тЬЕ implemented

*Archived roadmap item тАФ Element Ecosystem. Open items: [../ROADMAP.md](../ROADMAP.md) ┬╖ Index: [README.md](README.md)*

**Problem.** Homematic dimmers ramp: publish `LEVEL 0` while the dimmer is at 1 and it immediately reports back `0.95`, then `0.5`, тАж then `0` тАФ the slider the user just set **jumps around** during the ramp. Homematic exposes a **`WORKING` datapoint** (`true` while LEVEL is moving), but common interfaces (hm2mqtt, ccu-jack) **cannot guarantee** the `WORKING=true` message arrives *before* the first intermediate report тАФ only that it arrives shortly after (тЙд ~100 ms). RedMatic solves it differently: it additionally publishes **`тАж/LEVEL_NOTWORKING`** carrying only **settled** values (emitted once `WORKING` has returned to false). *(Blinds have the same ramp problem plus a `DIRECTION` datapoint тАФ split out to **E128**, later.)*

**Decided design (07/2026)** тАФ scope: the `*-light` family (brightness):

1. **Hold-at-target (own commands):** on publishing a set value the element enters **suppression immediately** тАФ this sidesteps the WORKING-ordering race entirely for user-initiated changes. The slider holds the user's target; incoming intermediate reports are swallowed until the reported value reaches the target, a settled signal arrives (`WORKINGтЖТfalse` or a `_NOTWORKING` message), or a **`settle-timeout`** (attribute, default 5 s) reconciles the slider to the last reported value (covers interrupted ramps / device clamping).
2. **`WORKING` topic attributes** (`WORKING` is a **distinct topic**, e.g. `hm/status/dimmer/WORKING`, mqtt-smarthome `{val: bool}` convention): `subscribe-working` + `message-property-working` (default `val`). `WORKING=true` enters/extends suppression (also for externally-initiated ramps); `WORKING=false` ends it and accepts the next report as settled.
3. **External-change race тЖТ ~100 ms display buffer:** incoming level reports render after a short delay (**`report-delay-ms`**, default `100`, `0` = off, only active when `subscribe-working` is wired); a `WORKING=true` arriving within the buffer cancels the pending jumpy update and enters suppression. Costs 100 ms latency on external changes only.
4. **RedMatic dual-topic mode:** new **`subscribe-settled`** attribute (e.g. `hm/status/dimmer/LEVEL_NOTWORKING`). When wired, **the slider position follows the settled topic only**, while the plain `LEVEL` subscription keeps driving the **numeric percentage readout live** (ramp feedback without handle jumping). Hold-at-target still applies after own publishes until the settled topic confirms.
5. **Discovery:** the Homematic recognizer (E108) already watches the firehose тАФ when it **observes** `LEVEL_NOTWORKING` for a channel (RedMatic publishes retained, so it shows up), the emitted config wires both topics (mode 4); otherwise it wires the channel's `WORKING` sibling topic (mode 2/3) when observed. No blind guessing тАФ only topics actually seen get wired.
6. **Implementation:** the settling machinery is a shared **family-agnostic helper** (shared module per the E106 pattern) тАФ built so **E128** can apply it to the cover family unchanged; тД╣ help texts explain the three wiring tiers (nothing / WORKING / settled topic). Family parity across material/glass/metro lights per **E114**.

**Ships with:** attributes + help texts across the light family (patch bumps), recognizer update + tests, TESTING.md notes (hold-at-target, timeout reconcile, external ramp with and without WORKING, RedMatic dual-topic, discovery wiring).

**Relates:** **E128** (blinds follow-up тАФ reuses the helper, adds `DIRECTION`), E108 тЬЕ (recognizer тАФ where `_NOTWORKING`/`WORKING` observation lands), **E114** (family parity contract), E102 тЬЕ-era climate work (same "device reports lag commands" family of problems тАФ setpoint shadowing there, level settling here), material/glass/metro light (the consumers), N37 (subscription lifecycle тАФ the settled/live dual subscription must play nice with pause/resume).

**Resolved (07/2026):** implemented as specced, commit `6eb27da3`. `SettlingController` lives in `@feezal/feezal-element/feezal-settling.js` (family-agnostic, 11 unit tests тАФ hold/timeout/sentinel/buffer/starvation-guard/settled-mode/dispose). Wired into material-, glass- and metro-light: `subscribe-working`/`subscribe-settled` + `message-property-*` twins (default `payload.val` per the mqtt-smarthome convention), `settle-timeout` (5 s), `report-delay-ms` (100); N6 inspectors gained a "Settling (Homematic ramps)" topics section and timeout/delay config fields; discovery maps `working_topic`/`settled_topic`. The Homematic light recognizer observes `WORKING`/`LEVEL_NOTWORKING` on the firehose and wires only topics actually seen (new recognizer test). One design refinement found by the existing E77 test: **sentinel targets (1.005 OLD_LEVEL) hold only when a WORKING/settled signal can end the hold** тАФ without one, the restore-ramp reports flow as before instead of freezing the ring for the timeout. Patch bumps: feezal-element 3.0.4, material-light 3.0.9, glass-light 3.0.16, metro-light 3.0.5. E128 (blinds + `DIRECTION`) stays open and reuses the controller unchanged.
