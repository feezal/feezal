**✅ Implemented (07/2026)** — exactly the extraction the item asked for:

- **`<feezal-theme-select>`** ([feezal-theme-select.js](../../www/src/feezal-theme-select.js)): shortened label + three-swatch row per option, trigger + dropdown chrome, an empty "Site theme (default)" entry with neutral swatches, a × clear on the trigger, user-theme ✏/delete affordances, and the **shared `sampleThemeColors()`** extraction (moved out of the sidebar — the sidebar lifts/restores its colour overrides around the call, since it owns them). `pkgToClass`/`pkgToLabel`/`DEFAULT_SWATCHES` exported from here.
- **Themes sidebar** mounts it (visual no-op — it was the donor); its old inline picker markup and swatch renderer are gone, `_sampleColors` delegates to the shared sampler (override placeholders keep working).
- **View inspector** mounts it via the E102-WP3 `type: 'custom', component:` descriptor hook (which had landed): the element mount reads/writes the view's `theme` attribute and emits `feezal-attribute-changed` through the normal dirty + undo pipeline — value `null` removes the attribute, preserving the **B50** contract (leading "Site theme (default)" entry + × clear). Bare-suffix and full-class attribute values both normalize.

Browser tests cover the component, both mount contracts and the B50 clear semantics; the stale U51 dropdown unit test was updated to the custom-mount assertion. TESTING.md gained the U53 row (B50's row updated to "picker").

Original item:

The U51 ✅ per-view `theme` attribute renders as a **plain generic dropdown** (full class names like `feezal-theme-solarized-dark`) in the view inspector, while the **site** theme selector in the themes sidebar is the nice one: **shortened names** (`pkgToLabel` strips the `feezal-theme-` prefix) plus **three colour swatches per theme** (base colours read from the theme's CSS custom properties, `DEFAULT_SWATCHES` for the built-in default). The two should be **exactly the same control**.

**Implementation:**
- **Extract the themes sidebar's option rendering into a reusable component** (e.g. `<feezal-theme-select>`): shortened label + swatch row per option, the swatch-extraction/caching logic (the sidebar already reads each theme's custom properties by temporarily applying it — reuse, don't duplicate), an empty option for "site theme" (view case) with neutral styling, and a `value`/`change` contract compatible with the inspector pipeline.
- **Themes sidebar** switches to the shared component (visual no-op there — it's the donor).
- **View inspector**: render the shared component instead of the plain dropdown — via the E102-WP3 `type: 'custom', component:` descriptor hook if that has landed. Emits through the normal change pipeline (dirty + undo) exactly like the dropdown did.
- Both places must stay in sync by construction afterwards — one component, two mounts (the E106 shared-building-block lesson applied to editor UI).

**Ships with:** TESTING.md update in the per-view-theme section (swatches + shortened names in the view inspector, selection still round-trips incl. × clear, sidebar unchanged visually).

**Relates:** **B50 ✅** (per-view theme bugs — the emptyOption entry + × clear and the layout-app shadow-root CSS mirroring; the new control preserves both), U51 ✅ (the per-view theme feature this polishes), U44 (× clear affordance — kept on the new control), E102-WP3 (the `type: 'custom'` inspector hook — the mount used), feezal-sidebar-themes.js (donor of label/swatch logic).
