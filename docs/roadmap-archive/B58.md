# B58 — *-climate: HmIP Manu mode ignored — `$setpoint` substitutes as a STRING in the putParamset payload ✅ fixed

*Archived roadmap item — Bugs. Open items: [../ROADMAP.md](../ROADMAP.md) · Index: [README.md](README.md)*

**✅ Fixed (07/2026)** — type-preserving substitution in all three families' `_resolveModePayload`: a value that IS the bare `'$setpoint'` sentinel becomes the **numeric** setpoint (so the putParamset object serializes as `{"CONTROL_MODE":1,"SET_POINT_TEMPERATURE":17}` — hm2mqtt accepts the FLOAT); the string replace remains only for embedded occurrences (`"temp=$setpoint"` → `"temp=17"`). BidCoS's plain-string `MANU_MODE=$setpoint` path publishes the same bare number string as before (top-level values go through `String()`). Browser-tested in all three families (JSON number type asserted, embedded-string path asserted). The helper remains triplicated — hoisting it is part of E137's `ClimateController` extraction. Part of the B53–B58 climate fix wave.

Original item:

Clicking **Manu** on an HmIP heating group does nothing — the thermostat stays in Auto. Superb user diagnosis with the wire evidence:

| Action | Published to `hm/paramset/<ch>/VALUES` | Device reaction |
|---|---|---|
| Manu | `{"CONTROL_MODE":1,"SET_POINT_TEMPERATURE":"17"}` | **ignored** (setpoint is a *string*) |
| Off | `{"CONTROL_MODE":1,"SET_POINT_TEMPERATURE":4.5}` | works (number) |

Replaying the Manu payload via `mosquitto_pub` with `17` unquoted works — **the JSON type is the whole difference**: hm2mqtt's putParamset needs the FLOAT parameter typed as a number; a string value is silently dropped.

**Root cause (confirmed by code read):** the E102 `$setpoint` placeholder is resolved by a **string replace** — `v => typeof v === 'string' ? v.replace(/\$setpoint/g, String(sp)) : v` (same helper duplicated in material/glass/metro per the keep-in-sync convention). The discovery-stamped Manu entry is `payload: {CONTROL_MODE: 1, SET_POINT_TEMPERATURE: '$setpoint'}` — the sentinel is a string, the replace *keeps* it a string, so the object serializes with `"17"`. The Off entry's `4.5` is a JSON **number literal**, never passes through `sub()`, stays a number — which is why Off works and Manu doesn't.

**Fix:** type-preserving substitution in all three families' shared helper — when a value is **exactly** the sentinel, substitute the numeric setpoint (`v === '$setpoint' ? sp : …`); keep the string replace only for embedded occurrences inside longer strings (e.g. a hand-wired `"temp=$setpoint"`). One-line change per family + a unit test asserting the published object carries a JSON number (and that BidCoS's string-payload `MANU_MODE=$setpoint` path — where the payload IS a plain string — still publishes the bare number string it does today).

**Acceptance:** HmIP: Manu click switches the thermostat to Manu at the remembered setpoint (wire shows an unquoted number); Off keeps working; BidCoS Manu unchanged; a hand-wired embedded-`$setpoint` string payload still substitutes textually.

**Relates:** E102 ✅ (the `$setpoint` machinery), E108 ✅ (discovery stamps the putParamset Manu entry), **E114** (the helper is triplicated — fix all three, ideally hoist it into the shared base while touching it), B53–B57 (the climate fix wave — this is the sixth member, and the most functionally severe: Manu is simply broken on HmIP).
