## 1. Repository layout

The repo uses **npm workspaces** inside `www/` to manage element and theme packages:

```
feezal/                         ← repo root
  bin/feezal.js                 ← thin entry point; delegates to server/bin/feezal.js
  Dockerfile                    ← multi-stage production image
  package.json                  ← root convenience scripts (install, test, start)
  scripts/
    generate-elements.js        ← CI helper: writes www/editor/feezal-elements.js
  server/                       ← "@feezal/feezal-server" npm package (the Node.js backend)
    bin/feezal.js               ← CLI entry point
    src/
      app.js                    ← Express application factory
      build/
        elements.js             ← element discovery + module generation
        build.js                ← legacy Rollup helper (export pipeline)
        export.js               ← static ZIP export
      ...
    dist/                       ← assembled at release time (not committed)
    package.json                ← name: "@feezal/feezal-server", files: ["bin/","src/","dist/"]
  www/                          ← Vite frontend (private, not published to npm)
    editor/
      index.html                ← editor SPA entry
      feezal-elements.js        ← generated by scripts/generate-elements.js
    src/                        ← editor and viewer UI components
    viewer-src/viewer.html      ← viewer HTML entry
    vite.config.js
    package.json                ← npm workspaces: ["packages/@feezal/*"]
    packages/@feezal/           ← all element and theme packages live here
  packages/
    create-feezal-element/      ← "create-feezal-element" npm package (CLI scaffolder)
```

The `www/package.json` declares `"workspaces": ["packages/@feezal/*"]`. After `npm install` inside `www/`, npm creates symlinks from `www/node_modules/@feezal/*` → `www/packages/@feezal/*`. The element discovery scanner (`server/src/build/elements.js`) and the Vite build resolve elements via those symlinks.

### What lives in git

- All source code (`server/src/`, `www/src/`, `www/editor/`, `www/viewer-src/`)
- All element and theme **source** packages (`www/packages/@feezal/*/`)
- Configuration files (`Dockerfile`, `vite.config.js`, `*.json`)
- CI workflows (`.github/workflows/`)
- `www/editor/feezal-elements.js` — committed as an empty/skeleton file; regenerated before builds

### What is NOT in git

- `server/dist/` — assembled at release time from `www/dist/` + element packages
- `www/dist/` — Vite build output
- `node_modules/` directories — installed locally, not committed

## 2. Development environment

### Prerequisites

| Tool | Version | Notes |
|---|---|---|
| Node.js | ≥ 18 (22 recommended) | Matches Docker base image |
| Docker | any recent | Only needed for Docker workflow |

### First-time setup

```sh
git clone https://github.com/feezal/feezal.git
cd feezal
npm install        # installs server deps, www deps, generates feezal-elements.js, and builds
```

The root `npm install` script runs the full chain:
1. `cd server && npm install` — server production + dev dependencies
2. `cd www && npm install` — frontend deps + creates `www/node_modules/@feezal/*` symlinks via npm workspaces
3. `node scripts/generate-elements.js` — generates `www/editor/feezal-elements.js`
4. `cd www && npm run build` — Vite production build

> **Note**: this chain is defined as the `"install"` lifecycle script in the root `package.json`, so it runs automatically whenever `npm install` is invoked at the root.

### Starting the dev server

Two terminals are needed — the Express backend and the Vite dev server:

**Terminal 1 — Express backend (port 3000)**
```sh
npm start
# expands to: node server/bin/feezal.js --www-dir ./www
```

**Terminal 2 — Vite dev server (port 5173)**
```sh
cd www
npm run dev
```

Open the editor at **http://localhost:5173/editor/** (Vite proxies API and Socket.IO calls to `localhost:3000`).

The Vite config already proxies `/editor/feezal-elements.js` and all `/api/`, `/socket.io/` paths to the Express server, so no CORS issues arise in dev.

### Running tests

```sh
# From the repo root:
npm test           # runs server tests (www has no tests yet)

# Directly:
cd server && npx vitest run
```

## 3. Versioning

### Git tags

Releases are triggered by pushing a **`v*` tag**:

```sh
git tag v1.2.0
git push origin v1.2.0
```

The Docker publish workflow fires on this event and pushes both a versioned tag and `latest` to GHCR. There is no separate "release branch" — `master` is always the release branch.

## 4. Building

### Frontend (Vite)

Before building, `www/editor/feezal-elements.js` must exist with the full element import list because `viewer-main.js` imports it at Vite build time to bundle all elements into `viewer-bundle.js`:

```sh
# Step 1 — generate the element import file
node scripts/generate-elements.js

# Step 2 — build
cd www && npm run build
# Output: www/dist/
```

The build produces:
- `www/dist/editor/index.html` — editor SPA (references elements at runtime via `/editor/feezal-elements.js`)
- `www/dist/viewer-bundle.js` — self-contained viewer bundle (all elements inlined, used by the static export)
- `www/dist/assets/` — JS/CSS chunks for the editor
- `www/dist/feezal-builtin-elements.json` — list of any built-in elements bundled directly into the editor chunk

### Assembling the dist for Docker

The Docker build (Stage 1) assembles `server/dist/` so the production image can serve everything from one directory:

```sh
cp -r www/dist/. server/dist/
mkdir -p server/dist/packages
cp -r www/packages/@feezal server/dist/packages/
```

Element packages are copied from the real source directory (`www/packages/@feezal/`) rather than from `www/node_modules/@feezal/` (which contains npm workspace symlinks). After the copy, `server/dist/` contains:
- `editor/`, `assets/`, `viewer-bundle.js` — Vite output
- `packages/@feezal/` — all element and theme packages

## 5. Releasing — Docker

### Automated

Pushing a `v*` tag triggers the **"Publish Docker image"** workflow (`.github/workflows/release-docker.yml`):

1. Sets up QEMU + Docker Buildx for multi-platform builds (`linux/amd64`, `linux/arm64`)
2. Logs in to **GitHub Container Registry (GHCR)** with `GITHUB_TOKEN` (no extra secret needed — OIDC-backed)
3. Extracts tags from the git ref:
   - `ghcr.io/feezal/feezal:<semver>` (e.g. `1.2.0`)
   - `ghcr.io/feezal/feezal:latest` (on default branch only)
4. Builds and pushes using GitHub Actions cache (`type=gha`)

### Multi-stage Dockerfile

The `Dockerfile` has two stages:

| Stage | Base | Purpose |
|---|---|---|
| `frontend-builder` | `node:22-alpine` | `git clone` from GitHub + `npm install` (server + www) + Vite build |
| production | `node:22-alpine` | Copies the full build output from stage 1 |

Stage 1 clones the repo directly from GitHub using the `GIT_REF` build arg (default `master`). This avoids NTFS junction issues when building on Windows. The CI workflow passes `GIT_REF=${{ github.ref_name }}` so releases build from the exact tag.

### Building locally

```sh
docker build -t feezal .
docker run -d --name feezal \
  -p 3000:3000 \
  -v feezal-data:/data \
  feezal
```

To build a specific tag: `docker build --build-arg GIT_REF=v1.2.0 -t feezal .`

### Environment variables supported by the image

| Variable | Default | Description |
|---|---|---|
| `FEEZAL_PORT` | `3000` | HTTP port |
| `FEEZAL_DATA` | `/data` | Data directory (persist this volume) |
| `FEEZAL_EDITOR_PASSWORD` | *(none)* | Plain-text password; hashed at startup |
| `FEEZAL_TRUST_PROXY_AUTH` | *(off)* | Trust proxy auth header (`x-auth-user`) |

## 6. CI (continuous integration)

The **"CI"** workflow (`.github/workflows/ci.yml`) runs on every push to any branch and on every pull request:

1. `npm install --prefix server` — server dependencies
2. `npm test --prefix server` — vitest unit + integration tests (covering API routes, storage adapter, topic matching)

No Vite build or lint step is run in CI — only the server-side code is tested there. The build is only run as part of the Docker release workflow.

## 7. Installing element packages

### Bundled elements (in the repo)

All official elements and themes live in `www/packages/@feezal/`. They are declared as workspace dependencies in `www/package.json` and npm workspaces symlinks them into `www/node_modules/@feezal/` after `npm install`. They are included in every release automatically.

### Scaffolding a new element

```sh
npm run create-element
```

This runs the scaffolder (`packages/create-feezal-element/index.js`). For elements intended to ship with feezal, place the new package in `www/packages/@feezal/` and add it to the `dependencies` in `www/package.json`. See [element-spec.md](element-spec.md) for the full authoring specification.

## 8. Required repository secrets

| Secret | Where used | How to obtain |
|---|---|---|
| `GITHUB_TOKEN` | Docker publish workflow | Provided automatically by GitHub Actions — no setup needed |

No other secrets are required. Docker publishing uses OIDC via `GITHUB_TOKEN` and the `packages: write` + `id-token: write` permissions declared in the workflow.
